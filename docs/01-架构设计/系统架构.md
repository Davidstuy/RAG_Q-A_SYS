# RAG 智能问答系统 - 架构设计

## 🎯 项目目标
构建一个基于 Java 生态的 RAG（Retrieval-Augmented Generation）智能问答系统，实现文档上传、语义检索和智能问答功能。

## 🔧 技术栈选择与理由

### 1. Spring Boot (应用框架)
**为什么选择：**
- ✅ **Java 生态集成**：你熟悉的 Java 技术栈，学习成本低
- ✅ **企业级成熟**：稳定可靠，适合生产环境
- ✅ **丰富的生态**：大量现成的组件和库
- ✅ **依赖注入**：便于管理和扩展组件
- ✅ **RESTful 支持**：天然支持构建 API

**适用场景：** 后端服务、微服务、企业级应用

---

### 2. LangChain4j (AI 框架)
**为什么选择：**
- ✅ **Java 原生**：专门为 Java 设计的 LangChain 版本
- ✅ **RAG 原生支持**：内置文档加载、分块、向量化流程
- ✅ **多模型支持**：OpenAI、Hugging Face、本地模型等
- ✅ **记忆管理**：内置对话历史管理
- ✅ **流式输出**：支持实时响应

**核心概念：**
- `DocumentLoader`：加载各种格式文档
- `TextSplitter`：智能分块（按段落、句子、token）
- `EmbeddingModel`：文本向量化
- `VectorStore`：向量数据库抽象
- `ChatLanguageModel`：LLM 对话模型

**适用场景：** AI 应用开发、RAG 系统、聊天机器人

---

### 3. Milvus/ChromaDB (向量数据库)
**为什么选择：**

**Milvus：**
- ✅ **高性能**：支持大规模向量检索（亿级向量）
- ✅ **多种索引**：IVF、HNSW 等多种索引算法
- ✅ **云原生**：支持 Kubernetes 部署
- ✅ **开源免费**：Apache 2.0 许可证
- ✅ **Java SDK**：有官方 Java 客户端

**ChromaDB：**
- ✅ **简单易用**：Pythonic API，学习成本低
- ✅ **轻量级**：适合小到中型项目
- ✅ **本地运行**：无需额外服务，方便开发测试

**选择建议：**
- 开发学习 → ChromaDB（快速上手）
- 生产环境 → Milvus（性能、扩展性）

**核心概念：**
- **向量（Embedding）**：文本的数值表示，相似文本向量距离近
- **余弦相似度**：衡量向量相似度的指标（0-1，越接近1越相似）
- **索引**：加速向量检索的数据结构
- **集合（Collection）**：存储向量和元数据的容器

---

### 4. OpenAI API (大语言模型)
**为什么选择：**
- ✅ **最强性能**：GPT-4/GPT-3.5 是目前最强大的通用模型
- ✅ **API 稳定**：成熟的接口，文档完善
- ✅ **上下文窗口大**：GPT-4 支持 128K tokens
- ✅ **性价比高**：按用量付费，适合初创项目
- ✅ **LangChain4j 原生支持**：开箱即用

**替代方案：**
- 国内：通义千问、文心一言、智谱 AI
- 开源：Llama 2、Mistral（需本地部署）
- 其他：Anthropic Claude、Google PaLM

## 🔄 核心工作流程

### 流程 1：文档上传与向量化

```
1. 用户上传 PDF/文档
   ↓
2. Spring Boot 接收文件
   ↓
3. 使用 Apache PDFBox 提取文本
   ↓
4. LangChain4j TextSplitter 分块
   - 默认：每块 500-1000 字符
   - 重叠：100-200 字符（避免语义断裂）
   ↓
5. OpenAI Embedding API 向量化
   - 文本 → 1536 维向量（text-embedding-ada-002）
   ↓
6. 存入向量数据库
   - 向量 + 原始文本 + 元数据（文件名、页码）
   ↓
7. 返回成功响应
```

**关键知识点：**
- **为什么需要分块？**
  - LLM 上下文窗口有限（不能一次性处理整个文档）
  - 提高检索精度（小块更聚焦）
  - 降低成本（Embedding 按字符计费）

- **为什么需要重叠？**
  - 避免重要信息在分块边界被切断
  - 保留上下文连贯性
  - 提高检索召回率

---

### 流程 2：智能问答

```
1. 用户提出问题
   ↓
2. Spring Boot 接收问题
   ↓
3. 问题向量化
   - 使用相同的 Embedding 模型
   ↓
4. 向量检索
   - 计算问题向量与文档向量的相似度
   - 返回 Top-K 最相关的文本块（默认 K=5）
   ↓
5. 构建 Prompt
   - 系统提示：定义 AI 角色
   - 检索到的文本：作为上下文
   - 用户问题
   ↓
6. 调用 OpenAI Chat API
   - GPT-4/GPT-3.5 生成回答
   ↓
7. 返回回答 + 引用来源
   - 包含引用的文档块
   - 显示相似度分数
```

**关键知识点：**
- **为什么需要向量化检索？**
  - 传统关键词搜索无法理解语义
  - 向量检索能找到"意思相近"的内容
  - 举例：搜索"苹果公司"，向量检索能返回关于"Apple Inc."的内容

- **什么是 Prompt Engineering？**
  - 设计合理的提示词，引导 LLM 生成期望的回答
  - 本项目中的 Prompt 模板：
    ```
    你是一个智能问答助手。请根据以下参考文档回答用户问题。
    如果参考文档中没有答案，请诚实说明。

    参考文档：
    {检索到的文本块}

    用户问题：{用户问题}

    请用中文回答，并标注引用来源。
    ```

---

### 流程 3：对话历史管理

```
1. 用户提出问题
   ↓
2. 从数据库/缓存加载历史对话
   - 存储格式：用户消息 + AI 回答 + 时间戳
   ↓
3. 构建完整上下文
   - 最近 N 轮对话（避免超出上下文限制）
   ↓
4. 在 Prompt 中包含历史
   ↓
5. LLM 基于历史和检索结果回答
   ↓
6. 保存新的一轮对话
```

**关键知识点：**
- **为什么需要对话历史？**
  - 支持多轮对话（追问、澄清）
  - 保持上下文连贯性
  - 提升用户体验

- **如何存储对话历史？**
  - 短期：Redis（快速访问）
  - 长期：关系数据库（MySQL/PostgreSQL）
  - 本项目：先用内存存储，后续升级为 Redis

## 💡 常见问题解答

### Q1: 为什么不直接用 Python 实现 RAG？
**A:**
- Python 生态确实成熟（LangChain、LlamaIndex）
- 但你的优势是 Java，用 Java 实现是差异化技能
- 很多企业后端是 Java，Java RAG 能无缝集成
- LangChain4j 已经相当成熟，功能不输 Python 版本

### Q2: 向量数据库可以不用吗？
**A:**
- 理论上可以用简单的内存存储 + 余弦相似度计算
- 但数据量大时性能差（O(n) 检索复杂度）
- 向量数据库使用索引，检索复杂度 O(log n)
- 生产环境强烈建议使用向量数据库

### Q3: OpenAI API 会被墙吗？
**A:**
- 中国大陆直接访问不稳定
- 解决方案：
  1. 使用代理（VPN/API 代理）
  2. 使用国内替代（通义千问、文心一言）
  3. 部署海外服务器

### Q4: 项目成本高吗？
**A:**
- OpenAI 按用量付费，相对便宜
- Embedding：$0.0001 / 1K tokens（约 750 英文单词）
- GPT-3.5：$0.002 / 1K tokens
- GPT-4：$0.03 / 1K tokens（输入）+ $0.06 / 1K tokens（输出）
- 估算：1000 次问答 ≈ $1-5
